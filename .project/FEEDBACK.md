## Результат рефакторинга `dashboard`

### Что выполнено

- Приведена структура к чистой многоуровневой архитектуре:
  - `domain` содержит бизнес-логику и нормализацию статусов;
  - `infrastructure` отвечает только за чтение/запись состояния dashboard;
  - `interfaces` обрабатывает HTTP и рендерит HTML;
  - `main` инициализирует зависимости и запускает сервер.
- Удалён дублирующий `dashboard/twsx.go`, источник twsx-классов только в `dashboard/pkg/twsx/twsx.go`.
- Введён `DashboardRepository` интерфейс и dependency injection в `NewDashboardService`.
- Добавлены единые доменные статусы (`running / stopped / paused`) и их нормализация для UI/статистики.
- Улучшена устойчивость API:
  - валидация методов и payload,
  - разбор и возврат ошибок через JSON,
  - корректные HTTP-коды (`400/404/405/500`).
- Добавлен `RegisterRoutes` для маршрутизации handlers.
- Написаны TDD-тесты (без внешних зависимостей): `domain`, `config`, `infrastructure`, `interfaces`, `pkg/twsx`.
- Приведены тестовые данные в `dashboard/data/dashboard.json` к согласованным статусам.

### Что ещё не входит в эту итерацию

- Реализация самого PaaS-уровня (`apps`, compose lifecycle, SSL, nginx, CRON/cleanup, безопасность) пока остаётся в следующем эпике.
- Текущий прототип остаётся полноценной платформенной панелью в рамках текущего sprint-среза, но не закрывает все требования production PaaS GUI из гайда.
