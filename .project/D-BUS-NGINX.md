Design is a Go **control plane** that treats systemd as the runtime and D-Bus as the management API, while NGINX remains the only ingress and route compiler for root `/` plus mounted sections like `/blog/`, `/catalog/`, and `/admin/`.  You do not need to replace systemd: systemd already exposes a central `Manager` object, per-unit objects, per-job objects, lifecycle methods such as `StartUnit`, `StopUnit`, `RestartUnit`, `Reload`, unit-file methods such as `EnableUnitFiles` and `DisableUnitFiles`, runtime mutation via `SetUnitProperties`, transient execution via `StartTransientUnit`, and signals such as `JobRemoved` and `UnitFilesChanged`, while `coreos/go-systemd` already provides Go packages for D-Bus, journal access, service notification, socket activation, and unit-file handling. [freedesktop](https://www.freedesktop.org/software/systemd/man/org.freedesktop.systemd1.html)

## Runtime boundary

Your panel should run as a single-user standalone root service and speak to the system bus, because the systemd service manager exposes its API on D-Bus and privileged operations are controlled there, while root avoids most interactive authorization friction.  Architecturally, the clean boundary is: Go owns desired state, release logic, templates, and orchestration plans; systemd owns process supervision, restart semantics, cgroups, and unit state; NGINX owns public URL routing and proxy behavior. [docs.nginx](https://docs.nginx.com/nginx/admin-guide/web-server/reverse-proxy/)

For control operations, prefer D-Bus calls over shelling out to `systemctl`, because systemd already models start/stop/restart/reload as manager methods and exposes asynchronous jobs and unit properties in structured form rather than plain text.  For deploy workflows, subscribe to D-Bus signals first, then execute lifecycle calls, and track completion through `JobRemoved`, because systemd documents that this is the race-free way to observe the actual result of a start or restart operation. [freedesktop](https://www.freedesktop.org/wiki/Software/systemd/dbus/)

## Project model

A good resource model is to make `Project` the top-level deployable object and `Service` the child runtime object, because systemd itself works on units while grouping is your control-plane concern, not a D-Bus limitation.  That means deploy, stop, restart, and delete stay at the `Project` level, but the planner expands those actions into ordered operations on the project’s child services and route fragments. [docs.nginx](https://docs.nginx.com/nginx/admin-guide/web-server/reverse-proxy/)

[docs.nginx](https://docs.nginx.com/nginx/admin-guide/web-server/reverse-proxy/) |

I would explicitly support two project modes: `single-site` for one main app at `/` with side services like PostgreSQL and Redis, and `sectioned-site` for one main app at `/` plus mounted helper services under prefixes like `/catalog/` or `/admin/`.  This switch belongs to your NGINX and resource-model policy, not to D-Bus, because systemd manages units by identity and state, while NGINX decides how URLs map to upstreams. [freedesktop](https://www.freedesktop.org/wiki/Software/systemd/dbus/)

## Go services

Split the backend into focused services: `ProjectService`, `DeployPlanner`, `GitManager`, `BuildRunner`, `ReleaseManager`, `UnitRenderer`, `SystemdManager`, `JournalReader`, `NginxRenderer`, and `Reconciler`.  This maps well to the available Go/systemd tooling, because `go-systemd` already has packages for `dbus`, `journal`, `sdjournal`, `daemon`, `activation`, and `unit`. [github](https://github.com/coreos/go-systemd)

`SystemdManager` should wrap `LoadUnit`, `StartUnit`, `StopUnit`, `RestartUnit`, `Reload`, `EnableUnitFiles`, `DisableUnitFiles`, `GetUnit`, `ListUnits`, `SetUnitProperties`, and optionally `StartTransientUnit` for ephemeral build, migration, or smoke-test jobs.  `JournalReader` should read logs through journald rather than scraping files, because `go-systemd` explicitly supports writing to and reading from the journal, including a dedicated `sdjournal` package for read access. [freedesktop](https://www.freedesktop.org/software/systemd/man/org.freedesktop.systemd1.html)

At deploy time, the planner should build a release graph per project, order services by dependency, write or update unit files, run a manager `Reload`, enable missing unit files, then start or restart the affected units and watch D-Bus jobs until they finish.  For short-lived one-off tasks such as DB migrations, asset compilation, or cache warming, use transient units instead of long-lived service files when the job should disappear after completion. [freedesktop](https://www.freedesktop.org/software/systemd/man/org.freedesktop.systemd1.html)

## NGINX design

Store `mountPath` as a normalized absolute prefix that always starts and ends with `/`, with `null` or empty meaning “not mounted” and `/` reserved for the main site service.  Validation should reject duplicates after normalization, reject non-public mounts on public services, and reject any attempt to claim `/` from more than one service in the same server block. [digitalocean](https://www.digitalocean.com/community/tutorials/nginx-location-directive)

For nested mounts like `/admin/` and `/admin/api/`, do not reject them automatically, because prefix routing can safely coexist if the generator makes precedence explicit and deterministic.  The simplest rule is: sort mounted locations by descending path length, emit `^~` prefix locations for generated proxy mounts, and reserve regex locations for hand-written profiles only, because location selection order matters and regex-based mixing is where ambiguity grows fastest. [digitalocean](https://www.digitalocean.com/community/tutorials/understanding-nginx-server-and-location-block-selection-algorithms)

For safe proxy generation, compile two pass modes from one field: `PreservePrefix` emits `proxy_pass http://127.0.0.1:9102;`, while `StripPrefix` emits `proxy_pass http://127.0.0.1:9102/;`, because NGINX treats `proxy_pass` with a URI differently from `proxy_pass` without a URI and uses that distinction to replace or preserve the matched location prefix.  Always emit `proxy_set_header Host $host`, `X-Real-IP $remote_addr`, `X-Forwarded-For $proxy_add_x_forwarded_for`, and `X-Forwarded-Proto $scheme`, because NGINX documents header rewriting as a normal part of reverse-proxy configuration. [nginx](https://nginx.org/en/docs/http/ngx_http_proxy_module.html)

Use these defaults by service type: SPA services should default to `StripPrefix` unless they are built to live under a subpath; API services can use either mode but usually want one stable contract and explicit upstream paths; ordinary helper microservices should start with `StripPrefix` unless the application is known to be route-aware under its mount prefix.  If a mounted app emits absolute redirects or cookie paths that assume `/`, the generator should optionally emit `proxy_redirect` and `proxy_cookie_path` rules, because NGINX provides both directives specifically to rewrite upstream redirects and cookie path metadata. [nginx](https://nginx.org/en/docs/http/ngx_http_proxy_module.html)

## Deploy flow

A project-level deploy should follow one plan: resolve project mode, load service graph, reserve ports, fetch code, build releases, render unit files, reload systemd, start or restart affected services over D-Bus, render NGINX config, test NGINX, and only then reload NGINX.  This ordering keeps process state and routing state consistent, because systemd gives you structured job tracking and NGINX expects validated config before a reload. [docs.nginx](https://docs.nginx.com/nginx/admin-guide/web-server/reverse-proxy/)

A minimal policy matrix is enough: `single-site` means one service owns `/` and others are internal or non-public; `sectioned-site` means one service still owns `/`, but selected child services additionally own mounted prefixes; `custom-profile` means the project points to a predefined NGINX profile when you want a specialized layout or a separate admin installation for unusual cases.  D-Bus does not block any of these modes, because it only concerns unit lifecycle and state, so the compatibility switch should live in your project config as `routingMode` or `nginxProfile`, not in the systemd adapter. [freedesktop](https://www.freedesktop.org/wiki/Software/systemd/dbus/)
